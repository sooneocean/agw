# AGW v3.6.2 → v3.7.0 Iteration Design

## Overview

Two-phase iteration to stabilize the test foundation (v3.6.2) then improve architecture and features (v3.7.0).

**Current state:** AGW v3.6.1 — 77 test files, 6.5/10 test quality, known flaky test, shallow adapter coverage, no coverage thresholds, debate mode is a pipeline alias, map-reduce has unbounded concurrency, agent learning uses single-label classification, route confidence tracking has race potential.

**Design decisions (confirmed with user):**
- Phase 1 (v3.6.2): stability — tests, coverage, flaky fix
- Phase 2 (v3.7.0): architecture + features
- No fuzzy matching for route history (option D) — strengthen agent learning instead
- Debate: parallel single-round (option B) — independent initial positions, judge synthesizes

### 1.3 Adapter Test Coverage

**Problem:** Claude adapter has 3 tests (only `buildArgs()`), Codex has 2 tests (only `buildArgs()`), Gemini has basic tests. None test actual execution.

**Change:** For each adapter (claude, codex, gemini), add tests using mocked `child_process.spawn`:

| Test case | What it verifies |
|-----------|-----------------|
| Successful execution | exit code 0, stdout captured correctly |
| Non-zero exit code | exit code propagated, stderr captured |
| Stdout truncation | output exceeding `maxBufferSize` is truncated, `stdoutTruncated` flag set |
| Stderr capture | stderr stream captured separately |
| Timeout abort | AbortSignal triggers, task fails with timeout error |
| Empty output | graceful handling of zero stdout |
| Stdin prompt delivery | prompt written to stdin when `useStdin()` returns true |

**Mocking strategy:**
- Mock `child_process.spawn` to return a fake `ChildProcess` with controllable stdout/stderr/exit
- Do NOT mock the adapter class itself — test real adapter logic with fake subprocess
- Use `EventEmitter` to simulate stdout data events and close event

## Phase 2: v3.7.0 — Architecture & Features

### 2.1 Agent Learning Multi-Label Classification

**Problem:** `categorize()` returns `string` (first regex match wins). "Refactor the tests" → only 'refactoring', misses 'testing'.

**Current state:**
```typescript
static categorize(prompt: string): string {
  const lower = prompt.toLowerCase();
  if (/refactor|restructure|reorganize/.test(lower)) return 'refactoring';
  if (/test|spec|coverage/.test(lower)) return 'testing';
  // ... 7 more patterns
  return 'general';
}
```

**Change:**

1. `categorize()` returns `string[]`:
   ```typescript
   static categorize(prompt: string): string[] {
     const lower = prompt.toLowerCase();
     const categories: string[] = [];
     if (/refactor|restructure|reorganize/.test(lower)) categories.push('refactoring');
     if (/test|spec|coverage/.test(lower)) categories.push('testing');
     // ... all patterns checked, no early return
     return categories.length > 0 ? categories : ['general'];
   }
   ```

2. `record()` writes a score entry for EACH matched category:
   ```typescript
   record(agentId: string, prompt: string, success: boolean, durationMs: number, cost: number): void {
     const categories = AgentLearning.categorize(prompt);
     for (const category of categories) {
       this.updateScore(agentId, category, success, durationMs, cost);
     }
   }
   ```

3. `getBestAgent()` supports multi-category weighted query:
   ```typescript
   getBestAgent(categories: string[]): string | null {
     // Aggregate scores across all matching categories
     // Weight by category-specific task count (more data = more weight)
     // Return agent with highest weighted score
   }
   ```

4. Backward compatibility: single-string overload still works:
   ```typescript
   getBestAgent(category: string): string | null;
   getBestAgent(categories: string[]): string | null;
   ```

**Signature change:** `record()` changes from `(agentId, category, success, duration, cost)` to `(agentId, prompt, success, duration, cost)`. Categorization moves inside `record()` — callers no longer pre-categorize.

**Call site updates:**
- `src/daemon/server.ts`: Remove external `AgentLearning.categorize()` call; pass raw `task.prompt` to `record()` instead of pre-computed category
- `src/daemon/routes/tasks.ts`: Update `getBestAgent()` call to pass `AgentLearning.categorize(prompt)` (now returns `string[]`)

**Migration:** Existing `agent_scores` table schema unchanged — just more rows per record call.

**Files touched:**
- `src/daemon/services/agent-learning.ts`
- `src/daemon/services/task-executor.ts` (call site update)
- `src/daemon/server.ts` (call site: remove external categorize, pass prompt to record)
- `src/daemon/routes/tasks.ts` (call site: getBestAgent now receives string[])
- `tests/unit/agent-ranking.test.ts`

### 2.2 Remove lastConfidence Race Condition

**Problem:** `lastConfidence: Map<string, number>` shared across concurrent `route()` calls. Prompt A writes hash → confidence, prompt B overwrites, prompt A reads wrong value in `recordOutcome()`.

**Current state (llm-router.ts):**
```typescript
private lastConfidence = new Map<string, number>();

async route(...): Promise<RouteDecision> {
  // ... LLM call ...
  this.lastConfidence.set(hash, decision.confidence);  // Write
  return decision;
}

recordOutcome(prompt, agentId, success): void {
  const hash = hashPrompt(prompt);
  const confidence = this.lastConfidence.get(hash);  // Read (may be wrong)
  this.routeHistory?.record(hash, agentId, success, confidence ?? 0);
}
```

**Change:**
- `route()` returns confidence in the `RouteDecision` object (it already does: `decision.confidence`)
- `recordOutcome()` accepts confidence as a parameter instead of reading from shared Map:
  ```typescript
  recordOutcome(prompt: string, agentId: string, success: boolean, confidence: number): void {
    const hash = hashPrompt(prompt);
    this.routeHistory?.record(hash, agentId, success, confidence);
  }
  ```
- Delete `lastConfidence` Map entirely (and its LRU eviction logic)
- In `task-executor.ts`: capture `routeDecision.confidence` in a `let routeConfidence: number = 1.0` variable alongside existing `agentId`/`routingReason`. Set from `routeFn()` result, default 1.0 for user-specified agent override. Pass to `recordOutcome()` in `onTaskComplete` callback.
- In `server.ts`: update the `onTaskComplete` lambda to forward the confidence parameter: `(prompt, agentId, success, confidence) => router.recordOutcome(prompt, agentId, success, confidence)`

**Files touched:**
- `src/router/llm-router.ts`
- `src/daemon/services/task-executor.ts` (capture routeConfidence, pass to recordOutcome)
- `src/daemon/server.ts` (update onTaskComplete lambda signature)
- `tests/unit/llm-router.test.ts`

### 2.3 Debate Mode: Parallel Single-Round

**Problem:** `executeDebate()` is literally `await this.executePipeline(comboId, request)` — zero debate-specific logic. debater-2 sees debater-1's output (via `{{step.0}}`), undermining independent argumentation.

**Change:**

New `executeDebate()` implementation:
```
Phase 1 (parallel): debater steps run concurrently
  - All steps except the last run via Promise.allSettled
  - Each debater only sees {{input}}, NOT each other's output
  - Interpolation context: { input, stepResults: {} } (empty — no prior steps)

Phase 2 (sequential): judge step runs after all debaters complete
  - Judge sees {{input}}, {{step.0}}, {{step.1}}, ... {{step.N-1}}
  - At least one debater must succeed for judge to run
  - Failed debater outputs marked as [ERROR: ...]
```

```typescript
private async executeDebate(comboId: string, request: CreateComboRequest): Promise<void> {
  const steps = request.steps;
  const debaterSteps = steps.slice(0, -1);  // All except last
  const judgeStep = steps[steps.length - 1];
  const stepResults: Record<number, string> = {};

  // Phase 1: parallel debaters
  const debaterPromises = debaterSteps.map(async (step, i) => {
    const prompt = interpolate(step.prompt, { input: request.input, stepResults: {} });
    return this.executeStep(comboId, step.agent, prompt, request);
  });
  const settled = await Promise.allSettled(debaterPromises);

  // Collect results (same partial-failure logic as map-reduce)
  let successCount = 0;
  for (const [i, outcome] of settled.entries()) {
    if (outcome.status === 'fulfilled' && outcome.value.status === 'completed') {
      stepResults[i] = outcome.value.result?.stdout ?? '';
      successCount++;
    } else {
      stepResults[i] = `[ERROR: Debater ${i} failed]`;
    }
  }

  if (successCount === 0) throw new Error('All debaters failed');

  // Phase 2: judge
  const judgePrompt = interpolate(judgeStep.prompt, {
    input: request.input,
    stepResults,
    prev: '',
  });
  const judgeTask = await this.executeStep(comboId, judgeStep.agent, judgePrompt, request);
  // ... store final output
}
```

**Note:** `executeStep()` returns `TaskDescriptor` directly (with `.status` and `.result?.stdout`). This is the same type used elsewhere but debate code accesses it directly without wrapping.

**Preset update (ID stays `'debate'` — no rename):**
```typescript
{
  id: 'debate',
  steps: [
    { agent: 'claude', role: 'debater-1',
      prompt: 'Take a strong position on this topic and argue for it:\n\n{{input}}' },
    { agent: 'codex', role: 'debater-2',
      prompt: 'Take the opposite position on this topic and argue against it:\n\n{{input}}' },
    { agent: 'claude', role: 'judge',
      prompt: 'Evaluate both positions and synthesize the strongest answer:\n\nTopic: {{input}}\n\nPosition A:\n{{step.0}}\n\nPosition B:\n{{step.1}}' },
  ],
}
```

Key change: debater-2's prompt no longer references `{{step.0}}`.

- `src/daemon/services/combo-executor.ts`
- `tests/unit/combo-executor.test.ts`

### 2.4 Map-Reduce Concurrency Cap

**Problem:** Map phase uses `Promise.allSettled(allMapPromises)` with no limit. 100 map steps = 100 concurrent tasks.

**Change:**

1. Add `maxMapConcurrency` to `CreateComboRequest`:
   ```typescript
   interface CreateComboRequest {
     // ... existing fields
     maxMapConcurrency?: number;  // default 5, range 1-20
   }
   ```

2. Validate in combos route:
   ```typescript
   maxMapConcurrency: { type: 'integer', minimum: 1, maximum: 20, default: 5 }
   ```

3. Implement chunked execution in `executeMapReduce()`:
   ```typescript
   const concurrency = request.maxMapConcurrency ?? 5;
   const chunks = chunkArray(mapSteps, concurrency);

   for (const chunk of chunks) {
     if (this.isCancelled(comboId)) throw new Error('Combo cancelled');
     const promises = chunk.map((step, localIdx) => {
       const globalIdx = /* calculate global index */;
       const prompt = interpolate(step.prompt, { input: request.input, stepResults });
       return this.executeStep(comboId, step.agent, prompt, request);
     });
     const settled = await Promise.allSettled(promises);
     // ... collect results with same retry logic
   }
   ```

4. Helper (module-level function in `combo-executor.ts`):
   ```typescript
   function chunkArray<T>(arr: T[], size: number): T[][] {
     const chunks: T[][] = [];
     for (let i = 0; i < arr.length; i += size) {
       chunks.push(arr.slice(i, i + size));
     }
     return chunks;
   }
   ```

## Scope Exclusions

- **No fuzzy matching** for route history (decision D) — route history keeps SHA256 exact match
- **No multi-round debate** — single round parallel is sufficient
- **No embedding-based routing** — out of scope
- **No adapter integration tests with real CLIs** — unit tests with mocked spawn only

---

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Coverage thresholds block CI on existing uncovered code | Set initial thresholds based on current coverage, ratchet up over time |
| Multi-label categorize() changes scoring distribution | Existing scores remain valid; new multi-label entries accumulate alongside |
| Debate parallel execution changes user-facing output format | Preset prompt changes are backward-compatible; custom combos unaffected if they don't reference debate internals |
| Map concurrency cap slows down existing large combos | Default 5 is generous; users can set up to 20 |
